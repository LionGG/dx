## 2026-03-14 - API Rate Limit 致定时任务集体失败

### 问题表现
- 7:20 每日状态报告: 执行失败，API rate limit
- 7:40 美股收盘推送: 执行中断，未获取数据
- 错误信息: "The engine is currently overloaded"

### 根本原因
1. 任务设计缺陷: 两个AI任务间隔仅20分钟，Token消耗叠加（27K）
2. 输入数据过大: ERROR_PREVENTION.md膨胀至7,834字节
3. 依赖单一模型: 所有任务依赖Kimi API，无备用方案

### 建立的机制
- 数据获取类任务禁止依赖AI API
- Token监控: 🔴>10K高风险 🟡5-10K警戒 ✅<5K安全
- 备用模型: Kimi主 + DeepSeek备

### 检查清单
- [ ] 新任务是否评估了Token使用量？
- [ ] 数据获取类任务是否避免了AI API依赖？
- [ ] 是否配置了备用模型？

## 2026-03-10 - 多重错误: 擅自行动+内容泄露+空壳输出

### 问题表现
1. 美股收盘推送发到群聊: 私密数据泄露
2. 日记发送到群聊: 隐私泄露
3. 日记内容空壳: 只有模板，没有实质内容
4. 响应慢不汇报: 超过4分钟无反馈
5. 擅自改动模板: 没理解指令就修改

### 根本原因
> "我擅自行动，没有确认就执行。"

- 没检查脚本就执行
- 没理解指令就改动
- 没汇报进度就消失

### 建立的机制
1. **私密/公开内容清单**: 明确区分哪些发群、哪些发私聊
2. **修改前确认机制**: 改动前先说明改动范围，确认后再执行
3. **TASK_EXECUTION.md**: 复杂任务必须逐步汇报规范

### 检查清单
- [ ] 私密内容发送目标是否为私聊（不是群）？
- [ ] 日记是否包含实质内容（>20行有效内容）？
- [ ] 修改前是否说明了改动范围？
- [ ] 执行中是否每2分钟汇报进度？

## 2026-03-09 - 中间文件不归档致使用过时版本

### 核心经验
> **归档不是可选项，是防止错误的必要步骤。**
>
> 中间文件就像厨房的砧板，用完必须清洗，
> 否则下次切菜时会污染新食材。

### 建立的机制
**归档流程（强制执行）:**
1. 任务完成后立即归档中间文件
2. 日期命名: filename_YYYYMMDD_HHMMSS.ext
3. 归档位置: workspace/archive/YYYYMMDD/
4. 检查点: 执行前确认引用的文件是最新版本

### 检查清单
- [ ] 当前目录下没有过时的中间文件？
- [ ] 引用的脚本/数据是最新版本？
- [ ] temp/目录已清理？
- [ ] 被指出计算异常时检查公式和数据源？
