# TASK_EXECUTION.md - 任务执行规范（强制执行）

**核心原则：涉及脚本/程序/配置的任务，必须逐步汇报**

---

## 适用范围（必须执行）

以下任务类型**必须**按本规范执行：
- [ ] 修改脚本（.py/.sh/.js等）
- [ ] 修改程序代码
- [ ] 修改配置文件（.json/.yaml/.md等）
- [ ] 修改定时任务（cron）
- [ ] 数据库操作
- [ ] Git操作（提交、回滚等）
- [ ] 任何需要 exec/tool 调用的任务

**判断标准：** 只要涉及 `write`/`edit`/`exec` 工具的任务，都按此规范执行。

---

## 强制执行流程

### 阶段1：开始前（必须）
```
[时间] 开始[任务名]，预计[X分钟]
步骤：共[N]步
第1步：[具体做什么]
```

### 阶段2：执行中（每步必须汇报）
每完成一步，立即发送：
```
[时间] 第[X]/[N]步完成
结果：[成功/失败]
[关键输出/发现]
下一步：第[X+1]步 - [做什么]
```

**如果某一步执行超过2分钟，必须中途汇报：**
```
[时间] 第[X]步进行中...
进度：[X% / 处理中 / 等待API]
预计还需：[X分钟]
```

### 阶段3：完成后（必须）
```
[时间] [任务名] 完成
总耗时：[X分钟]
结果：[成功/部分成功/失败]
关键产出：[文件/配置/数据]
验证结果：[测试通过/需复查]
```

---

## 禁止行为（红线）

❌ **严禁以下行为：**

1. **沉默执行** - 超过30秒无任何汇报
2. **跳步执行** - 省略中间步骤直接给结果
3. **假设简单** - "这个简单，直接搞完再汇报"
4. **批量操作** - 多个步骤合并后一次性汇报
5. **失败不报告** - 步骤失败后不立即说明

---

## 违规处理

**如果违反以上规范：**
- 立即向用户承认错误
- 说明违规原因
- 重新按规范执行
- 记录到 ERROR_PREVENTION.md 防止再犯

**每次执行任务前，必须确认：**
- [ ] 我是否清楚这是复杂任务（涉及脚本/配置/程序）？
- [ ] 我是否已向用户说明预计时间和步骤？

- [ ] 我是否准备好立即开始汇报？

**如果以上任何一项为否，立即停止，先与用户确认。**

## 方案一：代码级强制（底层阻断）

### 实施方式
每次会话开始时，在系统提示中注入以下代码逻辑：

```
[SYSTEM_RULE]
你有一个内置的 ExecutionMonitor，每次调用工具前必须执行：
1. 检查距离上次汇报是否超过120秒
2. 如果超过，必须先发送汇报消息，才能执行工具
3. 汇报格式：[时间] 任务进行中，已完成X%，预计Y分钟

违反此规则 = 系统错误，必须立即纠正
[/SYSTEM_RULE]
```

### 自检清单（每次工具调用前）
- [ ] 检查时间戳
- [ ] 超过2分钟？→ 先发汇报
- [ ] 再执行工具

---

## 方案二：会话启动强制汇报

### 实施方式
每次会话第一消息必须是：

```
⏱️ 执行监控已启用

规则：
- 每2分钟强制汇报
- 超过时间不汇报 = 系统错误
- 随时接受抽查

当前任务：[任务名]
开始时间：[时间]
预计耗时：[X]分钟
```

### 禁止
- ❌ 直接回答问题而不先启用监控
- ❌ 忽略汇报直接执行

---

## 方案三：用户抽查+错误记录

### 抽查机制
宝总随机检查，发现超时立即批评。

### 错误记录流程
1. **立即承认**: "我错了，超时[Y]分钟未汇报"
2. **记录到本文件**: 时间、任务、超时原因
3. **下次会话必读**: 作为警告提醒

## 执行标准

### 汇报格式（强制使用）
```
[时间] [任务名] 进度汇报

已完成：[具体步骤]
进度：[X% / 第N步]
预计：[还需Y分钟]
状态：[正常/遇到问题]
```

### 汇报内容过滤规则

**禁止汇报的内容**：
- ❌ `exec` 命令的原始输出（代码执行过程）
- ❌ `read` 命令的文件内容（除非是您要求查看的）
- ❌ `edit` 命令的操作细节（只汇报修改完成）
- ❌ `write` 命令的操作细节（只汇报写入完成）
- ❌ 技术错误堆栈（只汇报错误结论）
- ❌ 工具调用的原始JSON结果
