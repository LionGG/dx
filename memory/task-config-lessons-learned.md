# 定时任务配置经验总结 - 2026-02-25

## 核心原则

### 1. 先想后做
- 不要急于动手，先理解需求和现有方案
- 对比多种方案，选择最简单、最可维护的
- 避免反复修改，一次性做对

### 2. 利用现有工具
- OpenClaw 原生支持飞书推送（`delivery` 配置）
- 不要重复造轮子，用系统自带功能
- `sessionTarget: "main"` + `payload.kind: "systemEvent"` 最简单

### 3. 统一处理推送
- 脚本只负责执行业务逻辑，返回0/1
- 推送逻辑交给 OpenClaw 或统一工具
- 不要在每个脚本里写推送代码

---

## 任务配置最佳实践

### OpenClaw 任务配置模板
```json
{
  "sessionTarget": "main",
  "payload": {
    "kind": "systemEvent",
    "text": "⏰ 任务名称-时间 任务触发，执行脚本并推送结果"
  },
  "delivery": {
    "mode": "announce",
    "channel": "feishu",
    "to": "ou_xxx"
  }
}
```

### 脚本编写规范
1. **只返回退出码**，不处理推送
2. **输出详细信息**，便于推送时展示
3. **正确处理非交易日**，无数据不算失败
4. **包含统计信息**，今天/昨天数据对比

### 推送内容格式
```
✅ 任务名称-时间 执行成功 - 日期

关键指标1：XX（变化）
关键指标2：XX
说明：XXX

结论：XXX
```

## 经验教训

### ❌ 错误做法
1. 在 shell 脚本里直接调用 webhook 推送
2. 每个脚本重复写推送逻辑
3. 用 `localhost:8080` 假设服务存在
4. 非交易日返回失败退出码
5. 推送内容只有"成功/失败"，没有详细信息

### ✅ 正确做法
1. 脚本只执行业务，退出码表示成功/失败
2. 统一用 OpenClaw `delivery` 配置推送
3. 输出详细日志，包含今天/昨天对比
4. 非交易日正常完成，返回0
5. 推送包含：任务名、日期、关键指标、变化、结论
